# 个人网站 PRD

## 1. 项目背景

本项目是一个长期运营的个人品牌网站，用于集中展示个人介绍、教育与经历、项目、文章、作品成果、时间轴、联系方式、社交账号和简历下载。网站不能是一次性静态展示页，而应支持后续持续更新、内容扩展、多语言和迁移。

## 2. 项目目标

- 建立一个极简、克制、专业、有个人品牌感的内容型网站。
- 通过可视化后台管理主要内容，避免每次更新都修改代码。
- 同时考虑日本访问速度、中国大陆访问兼容、移动端和桌面端体验。
- 保证代码、内容、图片、域名和配置尽量可迁移。
- 控制长期维护成本、安全风险和人工维护负担。

## 3. 用户群体

- 访问者：同学、导师、招聘方、合作方、研究或项目关注者。
- 管理者：网站所有者本人，偏非技术使用，希望通过后台维护内容。
- 搜索引擎与社交平台：读取 SEO、Open Graph、结构化数据和多语言页面。

## 4. 使用场景

- 访问者快速了解身份、方向、经历和代表项目。
- 访问者阅读文章、研究记录、项目详情和作品成果。
- 访问者下载简历或通过邮箱、社交账号联系。
- 管理者新增项目、发布文章、上传图片、调整首页推荐内容。
- 管理者保存草稿、控制是否公开、修改 SEO 信息、导出和备份内容。

## 5. 页面结构

- 首页：品牌名称、定位、简介、代表项目、最新文章、代表经历、技能或研究方向、联系入口、社交账号、简历下载。
- 关于我：简介、教育、地区、关注领域、技能、语言、经历、理念、长期目标。
- 项目：项目列表与详情页。
- 文章：文章列表、分类/标签页、文章详情页。
- 作品与成果：文档、报告、图片、视频、演示稿、下载文件。
- 时间轴：教育、项目、事件、获奖、证书、公开活动。
- 联系：邮箱、社交账号、GitHub、隐私说明。
- 管理后台：建议路径 `/admin`。
- 系统页：404、RSS、sitemap.xml、robots.txt。

## 6. 功能需求

- 内容展示：项目、文章、作品、经历、教育、时间轴、分类、标签、媒体资源。
- 内容状态：草稿、发布、公开/隐藏、推荐到首页、排序。
- 内容关系：项目与标签，文章与分类/标签，相关文章，时间轴关联项目或经历。
- 文件能力：图片上传、封面图、附件、简历文件、作品下载文件。
- 首页配置：代表内容、区块顺序、文案、CTA、社交链接。
- SEO：页面标题、描述、OG 图片、canonical、自定义 URL、hreflang。

## 7. 后台需求

后台必须支持：

- 登录和管理员身份验证。
- 仪表盘。
- 项目、文章、作品、时间轴、经历、教育管理。
- 图片和附件管理。
- 网站设置、SEO 设置、菜单管理、首页内容管理。
- 草稿和发布状态。
- 数据导出与备份说明。
- 管理后台禁止搜索引擎收录。

推荐 CMS：Keystatic。原因见 `ARCHITECTURE.md` 和 `DEPLOYMENT_OPTIONS.md`。

## 8. 多语言需求

- 预留中文、日语、英语。
- 推荐 URL：`/zh/`、`/ja/`、`/en/`。
- 默认语言建议为中文，最终需由网站所有者确认。
- 未翻译内容可选择：回退默认语言、隐藏未翻译版本、显示“翻译准备中”。推荐对正式页面隐藏未翻译详情页，对列表页显示默认语言摘要需明确标记。
- 后台编辑方式：每个内容模型支持按语言填写字段，或按语言拆分内容文件并通过统一 `translationKey` 关联。

## 9. SEO 需求

- 自定义页面标题与描述。
- Open Graph 和社交分享图。
- Sitemap、robots.txt、canonical URL。
- 结构化数据：Person、Article、CreativeWork、Project 或 SoftwareSourceCode。
- RSS。
- 多语言 `hreflang`。
- 自定义 URL。
- 404 页面。
- 搜索引擎收录控制，后台不得收录。

## 10. 安全需求

- 管理后台必须有登录保护。
- 不在前端代码暴露密钥。
- 限制文件上传类型、大小和路径。
- 防范 XSS、CSRF、暴力登录、接口滥用、恶意文件、依赖风险。
- 环境变量不得提交到 Git。
- 备份与恢复流程清晰，支持误删除恢复。

## 11. 隐私需求

- 初期不收集访客敏感个人信息。
- 初期不开发用户注册、评论、社区、私信、投稿、支付、广告系统。
- 联系方式优先使用邮箱链接和社交账号。
- 不默认公开电话号码、家庭地址或高风险隐私信息。
- 如后续增加联系表单，必须设计垃圾信息防护、隐私说明和数据存储策略。

## 12. 性能需求

- 首页核心内容应快速显示。
- 图片自动压缩、响应式图片、懒加载。
- 字体、图标和关键静态资源本地部署。
- 减少前端 JavaScript，不加载无用第三方脚本。
- 移动端低带宽环境可用。
- Lighthouse 目标：Performance 90+，Accessibility 95+，Best Practices 95+，SEO 95+。
- Core Web Vitals 目标：LCP < 2.5s，CLS < 0.1，INP < 200ms。

## 13. 兼容性要求

- 支持现代 Chrome、Safari、Edge、Firefox。
- 支持主流 iOS 和 Android 浏览器。
- 前端不得依赖 Google Fonts、Google Analytics、Google Maps、Google reCAPTCHA、YouTube 嵌入等作为正常显示必需资源。
- 第三方服务必须有替代或降级方案。

## 14. 部署要求

- 至少支持三套部署路径：最低成本、中日访问平衡、中国大陆正式部署。
- 代码由所有者控制，Git 仓库由所有者控制。
- 配置通过环境变量管理。
- 可迁移到香港、日本、新加坡、中国大陆服务器。
- 可从一种 CMS 或托管平台迁移到另一种。

## 15. 维护要求

- 管理者通过后台可视化更新主要内容。
- 内容可导出、可备份、可恢复。
- 依赖定期检查，优先自动化。
- 不引入不必要数据库、微服务或复杂后台。

## 16. 备份要求

- Git 保存代码、内容文本、配置模板、CMS schema。
- 图片可初期进入 Git 或独立对象存储，最终取决于图片规模。
- 密钥、环境变量、个人证件、未公开隐私材料不得进入 Git。
- 支持完整导出内容和媒体。
- 有网站损坏后的恢复步骤。

## 17. 非目标

初期不做：

- 用户注册。
- 评论区。
- 社区。
- 私信。
- 用户投稿。
- 在线支付。
- 广告系统。
- 大型数据库后台。
- 实时互动功能。
- 高复杂度 3D、粒子或滚动劫持效果。

## 18. 验收标准

- P0 只开发极简测试项目列表/详情页，不开发正式首页、完整项目系统或文章系统。
- P0 必须验证 `/admin`、Keystatic、本地文件、Astro 读取、图片显示、构建和草稿过滤的完整链路。
- P0 使用 Keystatic local mode；P9 才进入 GitHub mode，真实密钥不得进入仓库或前端。
- 已明确推荐技术栈、CMS、部署路径、多语言架构、内容模型、安全方案、备份方案和路线图。
- 已单独记录假设和阻塞问题。
- 非技术用户能理解未来如何更新网站。
- 后续正式开发必须在网站所有者确认阶段 0 后进入。

## P2 范围

P2 交付首页、个人资料与关于我单页。首页包含 Hero、简介、关注方向、推荐项目、当前状态和联系 CTA；关于页包含公开资料、教育概览、兴趣、技能、语言、理念与未来方向。内容由 Keystatic singleton 管理，隐私字段默认关闭。P2 不开发完整项目系统、文章系统、作品系统、时间轴、多语言详情页或部署。
